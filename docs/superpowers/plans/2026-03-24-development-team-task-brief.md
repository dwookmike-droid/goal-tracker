# 골 트래커 개발팀 작업지시서

> 목적: 현재 프로젝트 우선 골 트래커를 실제 개발 가능한 작업 스트림으로 나누고, 리뷰팀/감사팀이 바로 감독할 수 있게 한다.

## 공통 원칙

- 홈의 첫 정보는 포커스 프로젝트다.
- 일 단위 기록은 `했다` 한 번으로 끝난다.
- 미기록은 자동 빈칸이다.
- 트래킹은 히트맵 + 회고형이다.
- 레거시는 과거 자랑이 아니라 현재를 미는 근거다.

## 개발 스트림

### 스트림 A: 홈 포커스 카드

범위:

- 포커스 프로젝트 조회
- 제목/진행률/다음 행동 표시
- `했다` 버튼 노출
- 레거시 응원 보조 영역

할 일:

- [ ] 홈 UI 상태 모델 정리
- [ ] 포커스 프로젝트 로딩 연결
- [ ] `했다` 액션 연결
- [ ] 기록 후 홈 상태 즉시 반영
- [ ] 빈 상태 처리

완료 기준:

- 홈에서 한 번의 탭으로 오늘 기록이 남는다

### 스트림 B: 일 기록과 히트맵

범위:

- 날짜별 `했다` 기록 저장
- 프로젝트별 히트맵 생성
- 주/월/년 단위 집계

할 일:

- [ ] 일 기록 데이터 모델 확정
- [ ] 중복 기록 정책 정의
- [ ] 주간 집계 함수 구현
- [ ] 월간 히트맵 데이터 변환 구현
- [ ] 연간 요약 데이터 변환 구현

완료 기준:

- 하나의 기록이 주/월/년 화면에 일관되게 반영된다

### 스트림 C: 회고/리캡

범위:

- 주간 리캡 문장
- 월간 리캡 문장
- 연간 리캡 문장
- 공유 카드용 요약 데이터

할 일:

- [ ] 리캡 데이터 구조 확정
- [ ] 주간 요약 규칙 구현
- [ ] 월간 요약 규칙 구현
- [ ] 연간 요약 규칙 구현
- [ ] 공유 카드용 뷰 모델 정의

완료 기준:

- 리캡 문장과 숫자가 같은 데이터 원천을 쓴다

### 스트림 D: 레거시 근거층

범위:

- 완료 프로젝트 저장
- 레거시 문장 생성
- 홈/회고에서 레거시 재사용

할 일:

- [ ] 완료 프로젝트 데이터 정리
- [ ] 레거시 요약 구조 정리
- [ ] 홈 응원 문구 매핑
- [ ] 회고와 공유 카드 연동 규칙 구현

완료 기준:

- 레거시가 현재 프로젝트를 밀어주는 근거로 연결된다

### 스트림 E: 온보딩 서브시스템

범위:

- 프로젝트 선택/생성
- 실행 밀도 설정
- 코칭 보조 톤 설정
- 포커스 프로젝트 지정
- 첫 `했다` 기록

할 일:

- [ ] 온보딩 상태 모델 정의
- [ ] 3문항 저장 구조 정의
- [ ] 포커스 프로젝트 확정 로직 구현
- [ ] 첫 `했다` 기록 연결
- [ ] 메인 홈 진입 연결

완료 기준:

- 온보딩 종료 직후 사용자가 이미 첫 기록을 가진 상태가 된다

## 리뷰팀 체크포인트

- [ ] 홈의 핵심 행동이 한 탭인지 확인
- [ ] `했다` 기록 후 데이터 반영이 즉시 보이는지 확인
- [ ] 히트맵이 단순 색칠이 아닌 실제 기록 기반인지 확인
- [ ] 리캡 문장이 과장 없이 데이터에 맞는지 확인
- [ ] 온보딩이 코치 앱처럼 보이지 않는지 확인

## 감사팀 체크포인트

- [ ] 현재 프로젝트 우선 원칙이 유지되는지
- [ ] 미기록이 실패 상태처럼 처리되지 않는지
- [ ] 레거시가 감성 장식으로 과장되지 않는지
- [ ] 범위를 벗어난 기능이 섞이지 않는지
- [ ] 스트림 간 의미 충돌이 없는지

## 작업 제출 규칙

- 스트림 단위로 PR 또는 변경 묶음 제출
- 제출 시 `변경 범위 / 사용자 영향 / 열어둘 질문` 3줄 요약 첨부
- 다른 스트림 문제를 같이 고치지 않는다

## 승인 게이트

### 게이트 1

- 홈 포커스 카드와 `했다` 기록이 동작한다

### 게이트 2

- 히트맵과 집계 데이터가 정확하다

### 게이트 3

- 리캡 문장과 레거시 근거가 맞물린다

### 게이트 4

- 온보딩이 포커스 프로젝트와 첫 기록까지 연결된다

## 금지 항목

- `했다` 외 추가 일일 입력 강제
- 빈칸을 실패로 저장
- 홈보다 코치 메시지를 더 크게 노출
- 레거시를 독립 피드처럼 분리
- 히트맵과 리캡이 서로 다른 기준 데이터를 사용
