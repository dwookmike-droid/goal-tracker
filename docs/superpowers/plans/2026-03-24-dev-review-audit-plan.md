# 골 트래커 개발·리뷰·감사 병렬 실행 문서

> **에이전트 작업용:** 이 문서는 `superpowers:subagent-driven-development` 또는 `superpowers:executing-plans`를 전제로 실행한다.  
> **운영 원칙:** 현재 프로젝트를 먼저 붙잡는 골 트래커를 유지하고, 홈 포커스 카드의 한 탭 `했다` 기록, 프로젝트 히트맵 + 회고형, 레거시는 현재를 미는 근거로만 쓴다.

**목표:** 개발팀, 리뷰팀, 감사팀이 서로 막지 않고 동시에 움직이되, 제품 방향이 흔들리지 않도록 감독 기준을 고정한다.

**핵심 판단 기준**

- 홈에서 현재 프로젝트가 가장 먼저 보여야 한다
- 한 탭 `했다` 기록이 트래킹의 시작점이다
- 트래킹은 히트맵과 회고로 누적 의미를 보여줘야 한다
- 레거시는 과거 자랑이 아니라 현재를 밀어주는 근거여야 한다
- 불필요한 범위 확장은 즉시 차단한다

---

## 공통 전제

- 동일 저장소에서 다른 작업자가 동시에 작업 중일 수 있다
- 다른 사람의 변경을 되돌리거나 덮어쓰지 않는다
- 기능 변경은 관련 파일만 수정한다
- 각 팀은 자기 산출물만 책임진다
- 최종 승인 전까지는 병합 전제의 추측 수정을 하지 않는다

---

## 개발팀 작업 스트림 분리

개발팀은 하나의 큰 작업으로 묶지 말고, 아래 스트림으로 분리한다. 각 스트림은 독립적으로 진행 가능해야 하며, 서로의 완료를 기다리지 않도록 설계한다.

### 스트림 A: 홈 포커스 카드

- 현재 프로젝트 우선 노출
- 한 탭 `했다` 기록 버튼 연결
- 상태 변화가 즉시 보이도록 함
- 레거시 요약은 보조 정보로만 배치

완료 기준:

- 홈 진입 시 현재 프로젝트가 즉시 식별된다
- 사용자가 한 번의 탭으로 수행 기록을 남긴다

### 스트림 B: 프로젝트 트래킹

- 프로젝트별 히트맵 표시
- 일별 `했다` 기록 누적
- 주간/월간/연간 회고형 요약 연결
- 기록 공백은 공백으로 보존

완료 기준:

- 사용자는 시간 축을 따라 자신이 해온 일을 읽을 수 있다
- 히트맵과 회고 문장이 같은 데이터를 기준으로 움직인다

### 스트림 C: 레거시 근거층

- 완료된 작업의 근거 데이터 정리
- 레거시 항목을 현재 프로젝트를 미는 근거로 연결
- 홈과 회고에서 재사용 가능한 근거 문구 제공

완료 기준:

- 레거시가 감성 장식이 아니라 현재 행동의 근거로 사용된다

### 스트림 D: 공통 상태/데이터 정합성

- 기록 단위, 프로젝트 단위, 회고 단위의 의미를 고정
- 빈 상태, 완료 상태, 진행 상태의 해석을 통일
- UI와 데이터 계층 간 용어를 맞춘다

완료 기준:

- 같은 상태를 화면마다 다르게 해석하지 않는다

## 리뷰팀 코드/기능 리뷰 체크포인트

리뷰팀은 "잘 만들었는가"보다 "제품 방향을 정확히 지켰는가"를 먼저 본다.

### 코드 리뷰 체크포인트

- 변경 범위가 한 스트림에 머무는가
- 불필요한 리팩터링이 섞이지 않았는가
- 상태 분기와 데이터 변환이 과도하게 복잡하지 않은가
- 새 코드가 기존 용어와 충돌하지 않는가
- 한 탭 `했다` 흐름이 여러 단계로 늘어나지 않았는가

### 기능 리뷰 체크포인트

- 홈에서 현재 프로젝트가 가장 먼저 보이는가
- `했다` 기록이 한 번의 행동으로 끝나는가
- 히트맵이 실제 기록 밀도를 정확히 반영하는가
- 회고형 화면이 누적 의미를 설명하는가
- 레거시가 현재를 밀어주는 문장이나 근거로 쓰이는가

### 리뷰 판정 기준

- 핵심 동작을 늦추는 디테일은 보류
- 핵심 동작을 흐리는 편의 기능은 반려
- 같은 의도를 가진 중복 구현은 하나로 축소

완료 기준:

- 리뷰 결과가 "방향 일치", "동작 단순성", "데이터 일관성"으로 요약된다

## 감사팀 범위/품질/데이터 의미 검증 포인트

감사팀은 구현의 정답 여부보다, 제품의 의미가 제대로 보존되는지를 검증한다.

### 범위 검증

- 이번 작업이 홈 포커스 카드, `했다` 기록, 히트맵, 회고, 레거시 근거 중 어디에 속하는지 확인
- 범위를 벗어난 기능이 섞였는지 확인
- 다음 단계로 미뤄야 할 항목이 현재 작업에 들어오지 않았는지 확인

### 품질 검증

- 기록 동작이 실패해도 데이터가 망가지지 않는가
- 빈 상태가 의미 없는 에러처럼 보이지 않는가
- 상태 전환이 사용자의 현재 맥락을 해치지 않는가
- 화면 간 문구와 숫자가 서로 어긋나지 않는가

### 데이터 의미 검증

- `했다`는 단순 체크가 아니라 수행 흔적인가
- 히트맵은 활동량의 시각화인가, 단순 색칠인가
- 회고는 요약이 아니라 변화 해석인가
- 레거시는 과거 자료가 아니라 현재를 밀어주는 근거인가

### 감사 결론 규칙

- 의미가 흔들리면 기능이 동작해도 보류
- 숫자가 맞아도 해석이 틀리면 보류
- 현재 프로젝트 우선 원칙을 훼손하면 보류

완료 기준:

- 감사팀이 "범위 적합", "품질 수용", "의미 보존" 세 가지를 통과시킨다

---

## 병렬 작업 운영 규칙

1. 개발팀은 기능 스트림을 쪼개서 동시에 진행한다.
2. 리뷰팀은 완성본만 기다리지 말고, 스트림 단위 산출물을 수시로 본다.
3. 감사팀은 코드 구현보다 먼저 의미와 범위를 검증할 수 있다.
4. 각 팀은 상대 팀의 결론을 선결 조건으로 삼지 않는다.
5. 단, 홈 포커스 카드의 핵심 문구와 `했다` 기록 의미는 세 팀이 공통으로 맞춘다.
6. 의사결정이 필요한 충돌은 한 문장으로 기록하고, 그 문장만 기준으로 다음 작업을 진행한다.
7. 긴 논쟁이 생기면 기능 추가가 아니라 범위 축소를 먼저 검토한다.

### 동시 작업 예시

- 개발팀 A가 홈 포커스 카드를 붙이는 동안
- 개발팀 B가 히트맵 데이터 바인딩을 정리하고
- 리뷰팀은 이미 끝난 파일의 변경 폭을 점검하고
- 감사팀은 `했다`와 회고의 의미 정의를 검토한다

이 방식의 목적은 속도가 아니라, 제품 방향을 유지한 채 병렬성을 확보하는 데 있다.

## 출시 직전 승인 게이트

출시 직전에는 아래 항목이 모두 통과해야 한다. 하나라도 실패하면 출시 보류다.

- 홈에서 현재 프로젝트가 가장 먼저 보인다
- `했다` 기록이 한 번의 탭으로 수행된다
- 프로젝트 히트맵이 실제 기록을 보여준다
- 회고형 화면이 기록을 해석한다
- 레거시가 현재를 미는 근거로 사용된다
- 범위를 벗어난 기능이 섞이지 않았다
- 리뷰팀이 방향 일치와 단순성을 승인했다
- 감사팀이 범위, 품질, 의미를 승인했다
- 다른 작업자의 변경과 충돌 없이 병합 가능하다

### 승인 문구

출시 직전에는 아래 세 문장을 모두 확인한다.

- "현재 프로젝트 우선 원칙이 유지되었다"
- "`했다` 기록이 핵심 경로로 남아 있다"
- "히트맵과 회고가 같은 의미를 말하고 있다"

모든 문장이 참이면 출시 승인, 하나라도 아니면 재작업이다.

## 최종 완료 기준

- 개발팀은 스트림별 산출물을 갖는다
- 리뷰팀은 코드/기능 리뷰 체크포인트를 통과시킨다
- 감사팀은 범위/품질/데이터 의미를 검증한다
- 병렬 작업 규칙이 실제 협업에 적용된다
- 출시 게이트가 명시적으로 통과된다
