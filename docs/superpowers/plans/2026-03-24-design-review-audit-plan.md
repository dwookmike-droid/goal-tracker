# 골 트래커 디자인 리뷰·감사 실행 계획

> **에이전트 작업용:** 이 계획을 실행할 때는 `superpowers:subagent-driven-development` 또는 `superpowers:executing-plans`를 사용한다. 단계 추적은 체크박스(`- [ ]`) 형식을 기준으로 한다.

**목표:** 골 트래커의 UX/UI를 `현재 포커스 프로젝트 우선`, `카드형/리스트형 선택`, `상태는 색+라벨+아이콘`, `트래킹은 프로젝트 히트맵+회고형`, `일 단위는 했다만 입력하고 미입력은 빈칸` 원칙으로 고정하고, 디자인팀 산출물과 리뷰·감사 절차를 한 번에 운영한다.

**구조:** 디자인팀은 화면과 컴포넌트 산출물을 만든다. 리뷰팀은 사용성, 정보 우선순위, 상태 표현, 전환 흐름을 검토한다. 감사팀은 규정 일관성, 범위 통제, 문서 충돌 여부, 승인 게이트 준수 여부를 검사한다. 홈, 트래킹, 공유 화면은 먼저 고정하고 나머지는 그 이후에 확장한다.

**기술 스택:** Figma, UX 와이어프레임, UI 컴포넌트 정의서, 상태/문구 규칙 문서, 공유 카드 레이아웃

핵심 책임:

- 목록 보기 전환 규칙 고정
- 상태 표현 규칙 고정
- 트래킹 히트맵과 회고 구조 고정
- 공유 카드의 정보 밀도와 톤 고정
- 리뷰와 감사의 책임 경계 분리

## 작업 원칙

- 홈은 현재 포커스 프로젝트를 가장 먼저 보여준다
- 목록은 카드형과 리스트형을 선택 가능해야 한다
- 상태는 색, 라벨, 아이콘을 함께 쓴다
- 트래킹은 프로젝트 히트맵과 회고형 구조를 쓴다
- 일 단위 기록은 `했다`만 입력하고, 입력이 없으면 빈칸으로 둔다
- 감성보다 우선순위와 해석 가능성을 먼저 지킨다
- 승인 전에는 시각 디테일을 늘리지 않는다
- 리뷰에서 통과하지 못한 항목은 감사 단계로 넘기지 않는다

## 작업 흐름 요약

### 1. 디자인팀 산출물 고정

- [ ] 홈 화면 프레임 확정
- [ ] 프로젝트 목록 카드형 프레임 확정
- [ ] 프로젝트 목록 리스트형 프레임 확정
- [ ] 트래킹 화면 프레임 확정
- [ ] 공유 화면 프레임 확정
- [ ] 상태 배지 규칙 확정
- [ ] 빈칸 상태와 `했다` 입력 상태 확정
- [ ] 공통 컴포넌트와 간격 규칙 확정

완료 조건:

- 화면마다 무엇이 1차, 2차, 3차 정보인지 구분된다
- 홈/트래킹/공유 화면이 같은 시각 언어를 공유한다
- 카드형/리스트형 전환 시 정보 구조가 흔들리지 않는다

### 2. 리뷰팀 검토

- [ ] 홈에서 포커스 프로젝트가 가장 먼저 읽히는지 확인
- [ ] 목록에서 카드형/리스트형 전환이 자연스러운지 확인
- [ ] 상태가 색만이 아니라 라벨과 아이콘으로도 식별되는지 확인
- [ ] 트래킹이 단순 체크가 아니라 프로젝트 흐름과 회고를 보여주는지 확인
- [ ] 일 단위 기록이 `했다` 한 번으로 끝나는지 확인
- [ ] 미입력 상태가 실패처럼 보이지 않고 빈칸으로 처리되는지 확인
- [ ] 공유 카드가 외부 노출에 충분히 요약적인지 확인

완료 조건:

- 사용자는 3초 안에 현재 프로젝트와 다음 행동을 이해한다
- 화면별 핵심 행동이 한 문장으로 설명된다
- 리뷰팀이 수정 요청을 우선순위별로 정리한다

### 3. 감사팀 점검

- [ ] UX 원칙과 화면 시안이 서로 충돌하지 않는지 확인
- [ ] 홈, 트래킹, 공유 화면이 우선순위에서 벗어나지 않는지 확인
- [ ] 상태 체계가 모든 화면에서 동일한 의미를 유지하는지 확인
- [ ] 빈칸 처리 규칙이 예외 없이 유지되는지 확인
- [ ] 트래킹의 히트맵 밀도와 회고 정보량이 과하지 않은지 확인
- [ ] 목록형과 카드형이 서로 다른 화면처럼 분리되지 않는지 확인
- [ ] 범위 밖 장식 요소가 추가되지 않았는지 확인

완료 조건:

- 디자인 변경이 규정, 범위, 일관성 기준을 동시에 만족한다
- 감사팀이 승인/보류/반려를 명확히 기록한다

## 디자인팀 작업

### 작업 1: 홈 화면

- [ ] 포커스 프로젝트 카드 구조 고정
- [ ] 현재 진행률 우선 배치
- [ ] 오늘의 다음 행동 우선 배치
- [ ] 빠른 `했다` 입력 진입점 배치
- [ ] 진행 중 프로젝트 흐름의 밀도 조절

완료 조건:

- 홈을 열면 현재 포커스 프로젝트가 가장 먼저 보인다
- 사용자는 추가 탐색 없이 오늘 할 일을 안다

### 작업 2: 프로젝트 목록

- [ ] 카드형 레이아웃 정의
- [ ] 리스트형 레이아웃 정의
- [ ] 보기 전환 토글 정의
- [ ] 상태 배지 규칙 정의
- [ ] 진행률과 제목의 정보 위계 정의

완료 조건:

- 프로젝트 수가 많아도 스캔이 빠르다
- 형태만 바뀌고 의미 구조는 바뀌지 않는다

### 작업 3: 트래킹 화면

- [ ] 프로젝트 히트맵 레이아웃 정의
- [ ] 주/월/년 전환 규칙 정의
- [ ] 회고형 요약 위치 정의
- [ ] 일 단위 빈칸과 `했다` 상태 정의
- [ ] 누적 흐름과 회고의 균형 정의

완료 조건:

- 사용자가 쌓아온 흐름을 한눈에 본다
- 회고가 시각적 보상으로 읽힌다

### 작업 4: 공유 화면

- [ ] 주간 공유 카드 정의
- [ ] 월간 공유 카드 정의
- [ ] 연간 공유 카드 정의
- [ ] 외부 공유용 문구 길이 규칙 정의
- [ ] 숫자, 제목, 회고 한 줄의 우선순위 정의

완료 조건:

- 카드가 짧고 명확하며 공유해도 맥락이 살아 있다

## 리뷰팀 검토 항목

- 홈의 1차 메시지가 포커스 프로젝트인지 확인
- 목록의 기본 스캔 경로가 카드형인지 확인
- 리스트형이 정보 과밀 없이 대체안으로 작동하는지 확인
- 상태가 색, 라벨, 아이콘으로 동시에 전달되는지 확인
- 트래킹이 단순 출석표처럼 보이지 않는지 확인
- `했다` 입력이 가장 낮은 마찰로 배치되는지 확인
- 미입력 상태가 불안, 실패, 경고처럼 보이지 않는지 확인
- 공유 카드가 앱 내부의 감성 요소를 과하게 복제하지 않는지 확인
- 각 화면의 CTA 개수가 과하지 않은지 확인

## 감사팀 체크 포인트

- 규정 일치: UX 원칙이 시안과 동일한지 확인
- 일관성: 상태와 색상 의미가 화면마다 다르지 않은지 확인
- 범위 통제: 홈, 트래킹, 공유 이외의 기능이 새로 생기지 않았는지 확인
- 우선순위: 홈 > 트래킹 > 공유 순서가 유지되는지 확인
- 입력 규칙: 일 단위 기록은 `했다`만 허용하는지 확인
- 빈칸 규칙: 미입력은 별도 실패 상태로 바꾸지 않는지 확인
- 전환 규칙: 카드형/리스트형 토글이 정보 의미를 바꾸지 않는지 확인
- 승인 추적: 승인/보류/반려 사유가 문서로 남는지 확인

## 승인 게이트와 리뷰 루프

### 게이트 0: 범위 잠금

- 홈, 트래킹, 공유 화면만 이번 범위로 고정
- `했다` 입력과 빈칸 처리 규칙 고정
- 카드형/리스트형 선택 규칙 고정

통과 조건:

- 추가 요구가 있더라도 이번 라운드에서 넣지 않는다

### 게이트 1: 구조 승인

- 홈 정보 위계 승인
- 트래킹 구조 승인
- 공유 카드 정보량 승인

통과 조건:

- 화면별 핵심 목적이 한 줄로 설명된다

### 게이트 2: 시각 승인

- 상태 배지 시스템 승인
- 간격, 타이포, 밀도 승인
- 카드형/리스트형 일관성 승인

통과 조건:

- 같은 의미의 정보가 화면마다 다른 방식으로 읽히지 않는다

### 게이트 3: 최종 감사 승인

- 규정 일치 확인
- 범위 초과 확인
- 예외 문구 확인
- 누락 상태 확인

통과 조건:

- 감사팀이 보류 사유 없이 승인한다

### 일일 리뷰 루프

1. 디자인팀이 당일 수정안을 제출한다
2. 리뷰팀이 사용성, 우선순위, 상태 표현을 즉시 검토한다
3. 감사팀이 규정과 범위만 점검한다
4. 당일 반영 여부를 승인 로그에 남긴다

운영 기준:

- 하루 안에 결론이 나지 않으면 범위를 줄인다
- 원칙 위반보다 시각 디테일을 우선하지 않는다

### 주간 리뷰 루프

1. 주 초에 이번 주 우선순위를 다시 고정한다
2. 중간에 홈, 트래킹, 공유 순으로 점검한다
3. 주말에 보류 항목을 다음 주로 넘길지 결정한다
4. 주간 승인 로그를 정리한다

운영 기준:

- 주간 리뷰는 결정을 늦추는 회의가 아니라 범위를 정리하는 장치다
- 다음 주로 넘길 항목은 반드시 사유가 있어야 한다

## 화면 우선순위

1. 홈
2. 트래킹
3. 공유
4. 프로젝트 목록
5. 기타 폴리시

### 홈 우선순위

1. 포커스 프로젝트
2. 오늘의 다음 행동
3. 진행률
4. 빠른 `했다` 입력
5. 진행 중 프로젝트 흐름

### 트래킹 우선순위

1. 프로젝트 히트맵
2. 주/월/년 전환
3. 회고 요약
4. 누적 흐름

### 공유 우선순위

1. 성취 한 줄
2. 핵심 숫자
3. 대표 회고
4. 공유 가능한 레이아웃 안정성

## 컷 기준

아래 항목은 이번 라운드에서 늦출 수 있다.

- 세부 마이크로 인터랙션
- 고급 애니메이션
- 추가 배경 장식
- 비필수 공유 템플릿
- 목록의 부가 필터

아래 항목은 컷하면 안 된다.

- 포커스 프로젝트 우선
- 카드형/리스트형 선택
- 상태의 색+라벨+아이콘
- 프로젝트 히트맵
- `했다` 입력
- 미입력 빈칸 처리

## 완료 기준

- 디자인팀 산출물이 화면별로 정리된다
- 리뷰팀이 검토 항목을 모두 통과 또는 보류로 명시한다
- 감사팀이 규정, 일관성, 범위 통제를 확인한다
- 홈/트래킹/공유 우선순위가 문서와 시안에서 동일하다
- `했다`와 빈칸 규칙이 모든 화면에서 일관된다
