# 골 트래커 UX/UI 팀 분업 실행 계획

> **에이전트 작업용:** 이 계획을 실행할 때는 `superpowers:subagent-driven-development` 또는 `superpowers:executing-plans`를 사용한다. 단계 추적은 체크박스(`- [ ]`) 형식을 기준으로 한다.

**목표:** 골 트래커의 핵심 UX/UI를 `현재 프로젝트 우선`, `히트맵 기반 트래킹`, `공유 가능한 리캡 카드` 방향으로 고정하고, 디자인팀과 기획팀이 병렬로 작업할 수 있게 한다.

**구조:** 디자인팀은 화면과 시각 시스템을 책임지고, 기획팀은 트래킹 규칙과 의미 체계를 고정한다. 두 팀은 같은 원칙을 공유하되 산출물은 분리한다.

**기술 스택:** 디자인 시안, UX 와이어프레임, 트래킹 규칙 문서, 공유 카드 구조

---

## 공통 원칙

- 현재 프로젝트가 홈 최상단
- 일 기록은 `했다`만 입력
- 기록이 없으면 자동 빈칸
- 주/월/년으로 갈수록 회고 비중 증가
- 상태는 색 + 라벨 + 아이콘
- 프로젝트 목록은 카드형/리스트형 선택 가능

## 디자인팀 작업

### 작업 1: 홈 구조 시안

- [ ] 포커스 카드 구조 확정
- [ ] 진행 중 프로젝트 흐름 영역 확정
- [ ] 레거시 응원 카드 위치 확정
- [ ] `했다` 버튼 위치 확정

완료 조건:

- 홈을 보는 순간 현재 프로젝트가 가장 먼저 읽힌다

### 작업 2: 프로젝트 목록 시안

- [ ] 카드형 목록 설계
- [ ] 리스트형 목록 설계
- [ ] 전환 토글 위치 설계
- [ ] 상태 배지 규칙 설계

완료 조건:

- 10개 이상 프로젝트도 빠르게 스캔 가능하다

### 작업 3: 트래킹 화면 시안

- [ ] 주간 블럭 화면 설계
- [ ] 월간 히트맵 화면 설계
- [ ] 연간 회고 화면 설계
- [ ] 범위 전환 UI 설계

완료 조건:

- 시간이 커질수록 사용자가 “내가 해왔다”는 느낌을 받는다

### 작업 4: 공유 카드 시안

- [ ] 주간 카드
- [ ] 월간 카드
- [ ] 연간 카드

완료 조건:

- 앱 밖으로 공유해도 보기 좋고 메시지가 명확하다

## 기획팀 작업

### 작업 1: 트래킹 규칙 정의

- [ ] `했다`의 기준 정의
- [ ] 빈칸 처리 규칙 정의
- [ ] 날짜별 기록 누적 규칙 정의
- [ ] 프로젝트별 집계 규칙 정의

완료 조건:

- 디자인과 개발이 동일한 기록 규칙을 공유한다

### 작업 2: 상태 체계 정의

- [ ] 포커스 정의
- [ ] 진행 중 정의
- [ ] 완료 정의
- [ ] 상태 전환 조건 정의

완료 조건:

- 화면마다 상태 해석이 달라지지 않는다

### 작업 3: 리캡 규칙 정의

- [ ] 주간 요약 문장 규칙
- [ ] 월간 리캡 문장 규칙
- [ ] 연간 회고 문장 규칙
- [ ] 공유 카드 대표 숫자 규칙

완료 조건:

- 리캡 문장이 짧고 명확하며 반복 가능하다

### 작업 4: 레거시 연결 규칙 정의

- [ ] 완료 시점의 레거시 입력 구조 정의
- [ ] 어떤 레거시가 응원에 사용되는지 정의
- [ ] 응원 문구 노출 위치 정의

완료 조건:

- 레거시는 감성 장식이 아니라 실제 근거로 작동한다

## 협업 방식

1. 기획팀이 규칙을 먼저 짧게 고정한다.
2. 디자인팀이 그 규칙 위에서 시안을 그린다.
3. 홈과 트래킹 화면은 같이 리뷰한다.
4. 공유 카드는 마지막에 묶어서 정리한다.

## 우선순위

1. 홈
2. 트래킹 화면
3. 프로젝트 목록
4. 공유 카드
5. 나머지 폴리시

## 오늘 바로 확정된 것

- 홈은 밸런스형
- 포커스 카드는 트래킹 중심
- 목록은 카드형/리스트형 선택
- 상태는 색 + 라벨 + 아이콘
- 트래킹은 히트맵 + 회고형
- 일 기록은 `했다`만 입력
- 기록이 없으면 자동 빈칸

## 완료 기준

- 디자인팀이 홈/목록/트래킹/공유 화면 시안을 가진다
- 기획팀이 트래킹/상태/리캡/레거시 규칙 문서를 가진다
- 두 문서가 서로 충돌하지 않는다
