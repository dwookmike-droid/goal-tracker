# 온보딩 코치 서브시스템 설계

## 제품 재정의

이 문서는 골 트래커의 온보딩만을 분리해 다시 설계한다.

핵심 전제는 분명하다.

- 이 제품의 주인공은 코치가 아니라 `현재 프로젝트`다.
- 코칭은 사용자가 시작을 미루지 않도록 돕는 `보조 레이어`다.
- 온보딩의 목표는 멋진 소개가 아니라 `포커스 프로젝트 지정`과 `첫 번째 했다 기록`까지 연결하는 것이다.

즉, 온보딩은 사용자를 설득하는 절차가 아니라
`지금 붙잡을 프로젝트를 정하고, 오늘의 첫 행동을 남기게 하는 짧은 진입로`여야 한다.

## 설계 원칙

1. 현재 프로젝트가 먼저다
온보딩에서 가장 먼저 확인해야 하는 것은 사용자가 지금 붙잡을 프로젝트다.

2. 코칭은 뒤로 물러난다
코치 캐릭터는 사용자를 주인공으로 세우는 말투만 제공한다.
설명과 선택의 중심은 항상 프로젝트와 실행이다.

3. 3문항은 유지하되 역할을 바꾼다
기존의 성격/응원 중심 질문이 아니라, 프로젝트 설정과 실행 조건을 수집하는 질문으로 바꾼다.

4. 첫 성공은 작아야 한다
온보딩 직후의 첫 행동은 부담 없는 `했다` 기록이어야 한다.
사용자는 계획을 다 짜는 것이 아니라, 이미 시작했다는 감각을 가져야 한다.

5. 화면마다 다음 행동이 명확해야 한다
각 화면은 설명보다 선택을, 선택보다 다음 화면 이동을 강하게 가져간다.

## 온보딩 흐름

### 전체 흐름

1. 화면 1 - 시작 인트로
2. 화면 2 - 3문항 중 1번: 현재 프로젝트 설정
3. 화면 3 - 3문항 중 2번: 실행 밀도 설정
4. 화면 4 - 3문항 중 3번: 코칭 보조 톤 설정
5. 화면 5 - 포커스 프로젝트 확정 및 첫 `했다` 기록
6. 완료 후 - 메인 홈으로 진입

### 플로우 목표

- 사용자가 지금 가장 중요한 프로젝트를 고르게 한다
- 그 프로젝트를 포커스 프로젝트로 지정한다
- 오늘 바로 할 수 있는 최소 행동을 선택하게 한다
- 첫 `했다` 기록으로 제품 사용의 문턱을 넘긴다

## 화면별 설계

### 화면 1: 시작 인트로

목적:

- 이 앱이 `프로젝트를 먼저 붙잡는 트래커`라는 점을 즉시 전달
- 코칭을 메인이 아닌 보조 역할로 안내
- 사용자가 자신의 현재 프로젝트를 떠올리게 만들기

핵심 카피 방향:

- 헤드라인은 프로젝트 중심이어야 한다
- 코치 언급은 보조 문장 1개 이하로 제한한다
- 감성보다 실행감이 먼저 와야 한다

예시 카피:

- 헤드라인: `지금 붙잡을 프로젝트부터 정하자`
- 서브: `코치는 옆에서 도울 뿐, 중심은 네가 밀고 있는 그 일이다.`
- CTA: `시작하기`

상태값:

- `screen_state = "intro"`
- `has_existing_projects = true | false`

디자인 메모:

- 배경은 차분하고 여백이 넓어야 한다
- 중심 시각 요소는 코치가 아니라 `하나의 프로젝트 카드` 또는 `프로젝트 묶음`이어야 한다
- 버튼은 단 하나의 주행동만 강조한다

### 화면 2: Q1 - 현재 프로젝트 설정

목적:

- 사용자가 지금 가장 먼저 붙잡을 프로젝트를 고르게 한다
- 신규 프로젝트 생성과 기존 프로젝트 선택을 한 화면 흐름으로 묶는다
- 포커스 프로젝트의 기준을 세운다

질문:

- `지금 가장 먼저 붙잡을 프로젝트는 뭐야?`

권장 선택지 구조:

- `기존 프로젝트에서 고르기`
- `새 프로젝트로 시작하기`
- `아직 모르겠어, 추천받기`

세부 동작:

- 기존 프로젝트에서 고르기를 선택하면 프로젝트 목록/검색으로 연결
- 새 프로젝트로 시작하기를 선택하면 프로젝트 제목 입력으로 연결
- 아직 모르겠어를 선택하면 최근 생성/진행 중 프로젝트를 우선 추천

저장값:

- `project_source = "existing" | "new" | "recommend"`
- `focus_project_id`
- `focus_project_name`

상태값:

- `screen_state = "project_select"`
- `selection_state = "empty" | "selected" | "searching" | "creating"`

디자인 메모:

- 이 화면은 추천보다 선택을 우선한다
- 목록은 과도한 설명 없이 제목 중심으로 보여준다
- 현재 프로젝트가 없는 경우에는 생성 흐름이 자연스럽게 이어져야 한다

### 화면 3: Q2 - 실행 밀도 설정

목적:

- 사용자의 실제 실행 가능량을 수집한다
- 첫 `했다` 기록의 부담을 낮춘다
- 이후 홈에서 보여줄 기본 행동 밀도를 정한다

질문:

- `이 프로젝트를 밀 때, 어느 정도 밀도로 가는 게 맞아?`

권장 선택지:

- `가볍게 - 하루 1개, 아주 작게`
- `보통 - 하루 2개, 꾸준히`
- `강하게 - 하루 3개, 몰입해서`

저장값:

- `execution_pace = "light" | "medium" | "strong"`

후속 사용:

- 첫 `했다` 기록 카드 수
- 홈의 오늘 행동 개수
- 기본 체크인 문구 강도

상태값:

- `screen_state = "pace_select"`
- `pace_state = "not_chosen" | "chosen"`

디자인 메모:

- 숫자는 설명보다 크고 읽기 쉽게 보여준다
- 선택지는 하루 단위 실행감으로 이해돼야 한다
- 이 질문은 코칭 성향이 아니라 행동 밀도를 묻는다

### 화면 4: Q3 - 코칭 보조 톤 설정

목적:

- 코치의 역할을 `주인공`이 아닌 `보조 톤`으로 고정
- 감정 위로보다 행동 유도를 위한 말투 차이를 선택하게 한다
- 이후 메시지의 어조를 일관되게 유지한다

질문:

- `코치는 어떤 방식으로 도와주면 좋을까?`

권장 선택지:

- `따뜻하게 정리해 주는 스타일`
- `담백하게 사실만 말하는 스타일`
- `조금 더 단호하게 밀어주는 스타일`

저장값:

- `coach_support_tone = "warm" | "neutral" | "firm"`

톤 해석:

- `warm`: 사용자의 망설임을 줄이고 진입을 부드럽게 만든다
- `neutral`: 감정 개입을 최소화하고 행동만 정리한다
- `firm`: 변명을 줄이고 바로 실행하도록 압축한다

상태값:

- `screen_state = "tone_select"`
- `tone_state = "not_chosen" | "chosen"`

디자인 메모:

- 코치 캐릭터를 전면에 내세우지 않는다
- 말투 차이는 카드 라벨과 짧은 예시 문장으로만 표현한다
- 각 톤은 모두 같은 목표를 향해야 한다. 차이는 분위기일 뿐 기능이 아니다

### 화면 5: 포커스 프로젝트 확정 및 첫 `했다` 기록

목적:

- 사용자가 선택한 프로젝트를 최종적으로 포커스 프로젝트로 고정
- 오늘 바로 가능한 최소 행동을 선택하게 함
- 첫 `했다` 기록으로 온보딩을 끝낸다

구성:

- 상단: `이 프로젝트를 오늘의 포커스로 둘까?`
- 중간: 프로젝트 요약 카드
- 하단 1: 포커스 확정 버튼
- 하단 2: 첫 `했다` 기록 카드

추천 카드 내용:

- 프로젝트 제목
- 한 줄 목표
- 오늘의 최소 행동 1개
- 예상 소요 시간

첫 `했다` 기록 방식:

- 사용자는 `했다` 버튼을 누르기만 하면 된다
- 선택된 최소 행동에 대해 설명 입력을 요구하지 않는다
- 메모는 선택 사항으로만 둔다

저장값:

- `is_focus_project = true`
- `first_done_record_id`
- `onboarding_completed_at`

상태값:

- `screen_state = "focus_and_first_done"`
- `done_state = "pending" | "recorded"`
- `focus_state = "pending" | "confirmed"`

디자인 메모:

- 포커스 확정과 첫 했다 기록은 하나의 연속된 행동처럼 느껴져야 한다
- 완료 축하보다 `다음에 또 할 수 있음`을 강조한다
- 이 화면에서 코치는 짧게 한 번만 말한다

## 3문항 재정의 요약

기존의 3문항은 아래처럼 바뀐다.

1. 현재 프로젝트 설정
- 무엇을 할지 고른다

2. 실행 밀도 설정
- 얼마나 자주 밀지 정한다

3. 코칭 보조 톤 설정
- 어떤 말투로 도울지 정한다

이 구조의 의도는 명확하다.

- 첫 번째 질문은 `무엇`
- 두 번째 질문은 `얼마나`
- 세 번째 질문은 `어떻게 도울지`

즉, 코칭이 아니라 실행 가능한 프로젝트 셋업이 중심이다.

## 톤 규칙

코칭 톤은 앱의 정체성이 아니라 보조 표현 방식이다.

### 공통 규칙

- 길게 설명하지 않는다
- 사용자를 평가하지 않는다
- 실패나 지연을 과장하지 않는다
- 결과보다 다음 행동을 더 크게 다룬다
- 사용자의 프로젝트 이름을 자주 호출해 맥락을 유지한다

### warm

- 목적: 진입 장벽을 낮춘다
- 어조: 부드럽고 짧게, 부담을 줄이는 방향
- 금지: 과잉 위로, 감정 몰입, 자기계발 문구 남발

### neutral

- 목적: 정보 전달을 가장 또렷하게 한다
- 어조: 담백하고 짧게, 사실과 다음 행동 중심
- 금지: 친근함을 과하게 덧씌우는 말투

### firm

- 목적: 미루는 경향을 끊고 실행을 앞당긴다
- 어조: 단호하지만 공격적이지 않게
- 금지: 비난, 압박, 죄책감 유도

## 화면별 상태 처리

### 인트로

- 프로젝트가 이미 있으면 `기존 프로젝트로 계속하기` 동선을 노출
- 프로젝트가 없으면 `새 프로젝트 시작하기`를 우선 노출

### 프로젝트 선택

- 선택 전에는 다음 버튼 비활성화
- 검색 중에는 최근 프로젝트 추천 영역 유지
- 신규 입력 중에는 입력값이 빈 상태여도 다음 동선 안내 제공

### 실행 밀도

- 선택된 값이 바로 요약 카드에 반영돼야 한다
- 사용자는 숫자보다 체감 문장으로 이해해야 한다

### 톤 선택

- 예시 문장은 한 줄만 제공
- 실제 앱 전역의 코치 카피는 이 값에만 종속된다

### 첫 했다 기록

- 체크 한 번으로 성공 상태 전환
- 성공 직후는 축하보다 다음 행동 제안이 우선
- 기록이 비어 있어도 온보딩은 끝낼 수 있어야 한다

## 디자인팀이 바로 써야 할 것

1. 화면 1 인트로
- 프로젝트 중심 헤드라인
- 코치 보조 레이어 카피

2. 화면 2 프로젝트 선택
- 기존/신규/추천 3분기 구조
- 프로젝트 카드 또는 검색 리스트

3. 화면 3 실행 밀도
- 3단계 카드형 선택
- 숫자보다 문장 우선

4. 화면 4 톤 선택
- warm / neutral / firm 3종 카드
- 톤 예시 1줄

5. 화면 5 포커스 확정 및 첫 했다
- 포커스 카드
- 첫 `했다` 버튼
- 온보딩 완료 상태

디자인 원칙:

- 코치는 시각적 주인공이 아니다
- 프로젝트 카드가 가장 큰 시각적 비중을 가져야 한다
- 첫 완료 경험은 가볍고 분명해야 한다

## 개발팀이 바로 써야 할 것

1. 온보딩 상태머신
- `intro -> project_select -> pace_select -> tone_select -> focus_and_first_done -> completed`

2. 저장 모델
- `focus_project_id`
- `project_source`
- `execution_pace`
- `coach_support_tone`
- `first_done_record_id`

3. 연동 규칙
- 포커스 프로젝트 지정 후 홈 진입 시 해당 프로젝트가 기본값이 된다
- 첫 `했다` 기록 후 홈의 오늘 상태가 활성화된다
- 코치 톤은 이후 모든 보조 문구에 재사용된다

4. 예외 처리
- 프로젝트가 없을 때는 신규 생성 동선을 열어둔다
- 사용자가 질문을 건너뛰지 못하도록 최소 선택은 강제한다
- 첫 `했다` 기록이 없어도 온보딩 완료는 가능해야 한다

## 제외 범위

이 문서에서는 아래를 다루지 않는다.

- 코치 AI 생성 로직
- 온보딩 이후 홈 화면 상세 설계
- 프로젝트 상세/트래킹 화면 전체 구조
- 알림 시스템의 세부 정책

이 문서의 목표는 온보딩을 `프로젝트 우선 + 첫 실행 연결` 구조로 고정하는 것이다.

## 최종 결론

온보딩의 최종 문장은 이것이다.

`코치는 옆에서 돕고, 나는 지금 이 프로젝트를 붙잡는다.`
