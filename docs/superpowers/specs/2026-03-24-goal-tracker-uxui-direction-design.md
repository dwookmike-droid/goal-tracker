# 골 트래커 UX/UI 방향 설계

## 제품의 존재 이유

이 앱은 목표를 적어두는 저장소가 아니다.

이 앱의 존재 이유는:

`사용자가 지금 가장 중요한 프로젝트를 놓치지 않고, 끝까지 밀어붙이게 만드는 것`

따라서 이 앱은 일반적인 생산성 도구처럼 차갑게 정보를 쌓는 것이 아니라,
프로젝트를 끝내는 흐름을 시각적으로 붙잡아 주는 개인 성장용 앱으로 설계해야 한다.

## 핵심 UX 원칙

1. 현재 프로젝트를 먼저 붙잡는다.
홈에서 사용자가 가장 먼저 보는 것은 포커스 프로젝트다.

2. 트래킹은 간단해야 한다.
일 단위 입력은 복잡한 평가가 아니라 `했다` 한 번으로 끝나야 한다.

3. 시간이 커질수록 보람이 커져야 한다.
주, 월, 년으로 갈수록 체크 기록이 아니라 누적된 흐름과 회고가 보여야 한다.

4. 레거시는 감성 장식이 아니라 근거다.
과거 완료 프로젝트는 현재를 응원하는 증거로 작동해야 한다.

5. 보기 좋은 화면은 공유 가능한 성취 카드로 이어져야 한다.

## 확정된 방향

### 1. 홈 구조

선택안:

- `B. 밸런스형`

이유:

- 포커스 프로젝트를 가장 먼저 보여주면서도
- 멀티 프로젝트 앱으로서의 전체 흐름을 유지할 수 있기 때문

홈 우선순위:

1. 포커스 프로젝트
2. 현재 위치와 진행률
3. 다음 행동
4. 전체 프로젝트 흐름
5. 레거시 응원

즉, 레거시는 주인공이 아니라 보조 증거다.

### 2. 포커스 카드 성격

확정:

- 감정 문장보다 `골 트래킹` 중심

카드 구성:

- 프로젝트 제목
- 진행률
- 오늘의 다음 행동
- 마일스톤 상태
- 빠른 `했다` 기록 버튼

### 3. 프로젝트 목록

확정:

- 사용자가 `카드형 / 리스트형`을 직접 선택
- 기본값은 `카드형`

이유:

- 프로젝트를 의미 있는 단위로 느끼게 하려면 카드형이 첫 인상에 유리함
- 하지만 프로젝트 수가 많아지면 리스트형 스캔이 더 빠름

### 4. 상태 표시 방식

확정:

- `색 + 라벨 + 아이콘`

이유:

- 색만으로는 해석이 불안정함
- 라벨은 즉시 이해를 돕고
- 아이콘은 반복 사용 시 스캔 속도를 높임

권장 상태:

- 진행 중
- 포커스
- 완료

### 5. 트래킹 구조

선택안:

- `프로젝트 히트맵 + 회고형`

이유:

- 단순 체크보다 시간의 흐름이 더 잘 보임
- 주/월/년 확장이 자연스럽고
- 보람, 회고, 공유 카드로 이어지기 쉬움

트래킹 단위:

- 일: 했다 / 빈칸
- 주: 7일 흐름
- 월: 히트맵 + 월 리캡
- 년: 누적 흐름 + 완료 프로젝트 회고

### 6. 일 단위 기록 방식

확정:

- 사용자는 `했다`만 누른다
- 입력이 없으면 자동으로 빈칸 처리

이유:

- 입력 마찰이 가장 낮음
- 실패 입력을 강요하지 않음
- 히트맵이 깔끔해짐

### 7. 기록 위치

확정:

- 기본 위치: 홈 포커스 카드
- 보조 위치: 프로젝트 상세

이유:

- 홈 한 탭이 가장 낮은 마찰
- 상세 화면은 부가 편집/확인용

## 화면별 UX 목적

### 홈

목적:

- 오늘 무엇을 밀어야 하는지 즉시 이해
- 한 탭으로 `했다` 기록
- 전체 프로젝트 흐름도 놓치지 않음

핵심 컴포넌트:

- 포커스 카드
- 진행 중 프로젝트 흐름
- 레거시 응원 카드

### 프로젝트 목록

목적:

- 여러 프로젝트를 빠르게 스캔
- 카드형/리스트형 보기 전환
- 상태와 진행률을 명확하게 파악

### 프로젝트 상세

목적:

- 프로젝트의 맥락과 목표를 확인
- 진행률과 마일스톤 관리
- 포커스 지정 및 완료 처리

### 트래킹 화면

목적:

- 내가 실제로 얼마나 이어왔는지 시각적으로 확인
- 범위가 커질수록 보람을 느끼게 함

구조:

- 주: 7일 블럭
- 월: 히트맵 + 이번 달 리캡
- 년: 누적 흐름 + 완료 프로젝트 수 + 대표 회고

### 공유 화면

목적:

- 성취를 예쁘게 요약
- 사용자가 스스로 뿌듯함을 느끼게 함
- 외부 공유 가능

권장 유형:

- 이번 주 리캡 카드
- 이번 달 리캡 카드
- 올해 프로젝트 리캡 카드

## 비주얼 방향

레퍼런스 성격:

- 따뜻한 아이보리 배경
- 민트/코랄 포인트
- 둥근 카드
- 에디토리얼 느낌의 강한 타이포
- 정보량을 줄인 집중형 구성

단, 감성에 과하게 기울지 않고
`트래킹 중심의 담백함`을 반드시 유지해야 한다.

즉 시각 톤은 따뜻하지만 기능 우선순위는 매우 선명해야 한다.

## 디자인팀 역할

디자인팀은 `보이는 경험`을 책임진다.

필수 산출물:

1. 홈 화면
- 포커스 카드
- 진행 중 프로젝트 흐름
- 레거시 응원 카드

2. 프로젝트 목록
- 카드형
- 리스트형
- 보기 방식 전환 UI

3. 프로젝트 상세
- 진행률
- 마일스톤
- 포커스 지정
- 완료 버튼

4. 트래킹 화면
- 주/월/년 전환
- 히트맵
- 월 리캡
- 년 리캡

5. 공유 카드
- 주간 리캡
- 월간 리캡
- 연간 리캡

디자인팀 체크리스트:

- 상태 표현은 색 + 라벨 + 아이콘
- 포커스 카드의 1차 정보는 제목, 진행률, 다음 행동
- 레거시 카드는 메인이 아니라 보조 위치
- 카드형/리스트형 전환 시 시각 규칙 일관성 유지

## 기획팀 역할

기획팀은 `논리와 구조`를 책임진다.

필수 산출물:

1. 트래킹 규칙 정의
- `했다`의 의미
- 빈칸 처리 규칙
- 주/월/년 집계 규칙

2. 상태 체계 정의
- 진행 중
- 포커스
- 완료
- 이후 확장 상태가 필요한지 판단

3. 리캡 문장 규칙 정의
- 주간 요약 문장
- 월간 요약 문장
- 연간 회고 문장

4. 레거시 연결 규칙 정의
- 어느 시점에 레거시 입력을 받는지
- 어떤 문장이 현재 프로젝트 응원에 사용되는지

5. 공유 카드 정보 우선순위 정의
- 어떤 숫자를 크게 보여줄지
- 어떤 문장을 대표 문장으로 쓸지

기획팀 체크리스트:

- 입력 마찰을 늘리는 항목 금지
- 감성보다 행동 완료율을 우선
- 회고는 길지 않고 한눈에 읽혀야 함
- 공유 카드는 기능보다 자기 보람을 우선

## 최종 결론

이 앱은 결국 아래 문장을 UX로 번역하는 작업이다.

`나는 지금 이 프로젝트를 밀고 있고, 그 흐름이 시간 위에 쌓이고 있다.`

그래서 골 트래커 UX/UI의 핵심은

- 현재 프로젝트를 붙잡는 홈
- 간단한 일 기록
- 점점 커지는 리캡 보람
- 공유하고 싶은 결과 화면

이 네 가지에 집중해야 한다.
